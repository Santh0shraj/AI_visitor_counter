# Face Tracker

## 1. Project Overview
This project is an automated face tracking and recognition system designed to monitor and catalog unique visitors from a video stream. It leverages modern deep learning models for detection and feature extraction to maintain accurate session and historical counts of individuals. A lightweight SQLite database ensures data persistence while an optional Flask dashboard provides real-time traffic statistics.

## 2. Architecture
- **`detector.py`**: Utilizes YOLOv8 to locate and extract precise bounding boxes for faces within a video frame.
- **`recognizer.py`**: Uses InsightFace (ArcFace) to generate 512-dimensional structural embeddings from the cropped face regions and computes cosine similarities.
- **`tracker.py`**: A custom, lightweight state engine that maintains active records of faces across consecutive frames to determine when visitors enter or leave the view.
- **`database.py`**: Manages all SQLite interactions using WAL mode, storing encoded facial embeddings and logging entry/exit events safely.
- **`logger.py`**: Handles generating formatted text logs for the console and log files, as well as cropping and saving physical snapshot images of faces upon entry and exit.
- **`counter.py`**: Tracks in-memory session counts of recognized visitors while interfacing with the database layer to retrieve absolute global unique visitor metrics.
- **`main.py`**: The central controller loop that integrates the components, manages configuration loading, handles video framing, coordinates inference, and orchestrates the tracking logic safely.

## 3. Setup Instructions
- `pip install -r requirements.txt`
- Place your video file as `sample.mp4` in the project root
- `python main.py` to run
- `python app.py` to run the optional dashboard

## 4. config.json explained

| Key | Type | Description |
|---|---|---|
| `video_source` | String | Path to the input video file (e.g., `"sample.mp4"`). |
| `frame_skip_interval` | Integer | Specifies how many frames to skip between expensive detection inferences to save compute. |
| `similarity_threshold` | Float | The minimum cosine similarity score required to classify two embeddings as the same person. |
| `exit_timeout_frames` | Integer | The number of consecutive absent frames before an active track is formally considered "exited". |
| `log_dir` | String | Root directory specifically for storing text logs (`events.log`) and image crop folders. |
| `db_path` | String | The persistent file path pointing to the SQLite tracking database. |
| `detection_confidence` | Float | The minimum confidence score generated by YOLO to accept a bounding box as a valid face. |

## 5. Sample config.json
```json
{
  "video_source": "sample.mp4",
  "frame_skip_interval": 3,
  "similarity_threshold": 0.45,
  "exit_timeout_frames": 30,
  "log_dir": "logs",
  "db_path": "db/faces.db",
  "detection_confidence": 0.5
}
```

## 6. Assumptions made
- YOLOv8n-face model is used for detection
- InsightFace ArcFace model is used for recognition
- Cosine similarity threshold of 0.45 is used to distinguish faces
- SQLite WAL mode ensures crash-safe writes
- A face is considered exited if unseen for `exit_timeout_frames` consecutive frames

## 7. Compute load estimates

| Module | CPU Usage | GPU Usage | Notes |
|---|---|---|---|
| `FaceDetector` (YOLO) | Medium | High | Performs dense spatial convolutions on full frames. Skip intervals heavily mitigate load. |
| `FaceRecognizer` (InsightFace) | High | Low (CPU Provider) | Set to `CPUExecutionProvider` specifically to decouple from GPU bounds, but generates heavy CPU processing during dense crowds. |
| `FaceTracker` | Low | None | Simple mathematical dictionary lookups and bounds checks. |
| `DatabaseManager` | Low | None | SQLite writes efficiently on Disk IO, especially configured via WAL mode transactions. |
| Images / Logger | Low | None | OpenCV array cropping and JPEG encodes minimally utilize CPU. |

## 8. AI Planning and Architecture
The system was designed with strict modularity in mind, separating deep learning inferences from business logic and data storage. By isolating the complex tracking elements into simple, testable classes (`DatabaseManager`, `Logger`, `Tracker`), the `main.py` loop remains procedural and straightforward. Memory efficiency was heavily prioritized—employing frame skipping, strict exception handling, bounding box buffering, and localized try/excepts to ensure a bad frame or missing detection never crashed the entire application backbone.
